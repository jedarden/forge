# Control Panel TUI Dashboard Design

**Design Date**: 2026-02-07
**Framework**: Textual
**Purpose**: Interactive real-time dashboard for control panel monitoring and control

---

## Overview

The Control Panel Dashboard is a terminal-based user interface that provides real-time visibility into:
- Worker pool status and distribution
- Subscription usage and limits
- Task queue and bead status
- Cost analytics and optimization recommendations
- System activity and errors

The dashboard enables interactive control:
- Spawn/kill workers by model type
- Adjust pool targets and allocations
- Pause/resume optimization
- View detailed worker logs
- Manual task assignment

---

## Design Principles

1. **Information Hierarchy** - Most critical info (worker health, subscription limits) at top
2. **Glanceable** - Status should be understandable in 1-2 seconds
3. **Color-coded** - Use color to convey status (green=healthy, yellow=warning, red=critical)
4. **Progressive Disclosure** - Summary view by default, details on demand
5. **Keyboard-driven** - Efficient navigation with keyboard shortcuts
6. **Real-time** - All displays update live (1-2 second refresh)
7. **Responsive** - Adapts to terminal size

## Interaction Flows

### Spawn Worker Flow

1. User presses 'W' to open Worker Detail View
2. User presses 'G' to spawn GLM worker
3. Modal appears: "Spawn GLM Worker - Select Workspace:"
4. User selects workspace from list or enters path
5. System spawns worker, shows confirmation
6. Worker appears in table within 2-3 seconds
7. Activity log shows spawn event

### Kill Worker Flow

1. User navigates to Worker Detail View ('W')
2. User selects worker with arrow keys
3. User presses 'K' to kill
4. Confirmation modal: "Kill worker-glm-01? [Y/N]"
5. User confirms with 'Y'
6. Worker status changes to "Terminating"
7. Worker disappears from table within 5 seconds
8. Activity log shows kill event

### Task Assignment Flow

1. User navigates to Task Queue View ('T')
2. User selects ready task with arrow keys
3. User presses 'A' to assign
4. Modal shows available idle workers
5. User selects worker or chooses "Auto-assign"
6. Task moves to "In Progress" section
7. Worker table updates to show assignment
8. Activity log shows assignment event

## Data Update Strategy

### Update Frequencies

- Worker status: 2 seconds
- Subscription usage: 5 seconds
- Task queue: 3 seconds
- Activity log: Real-time (event-driven)
- Cost analytics: 10 seconds
- System health: 1 second

### Update Implementation (Textual)

```python
# Reactive variables for automatic UI updates
active_workers = reactive(0)
subscription_usage = reactive({})
ready_tasks = reactive([])

# Background async tasks
async def on_mount(self):
    self.run_worker(self.monitor_workers())
    self.run_worker(self.monitor_subscriptions())
    self.run_worker(self.monitor_tasks())
    self.run_worker(self.listen_activity_events())

async def monitor_workers(self):
    while True:
        data = await pool_manager.get_worker_status()
        self.active_workers = data.active_count
        self.update_worker_table(data.workers)
        await asyncio.sleep(2)
```

### Event-Driven Updates

For activity log and critical status changes:
- Use message passing from pool manager to dashboard
- Immediate UI update on event receipt
- No polling for real-time events
